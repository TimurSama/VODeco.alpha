# Токеномика и экономика VODeco (канон 2026)

## 1) Главный принцип
**Ценовая якорная единица — вода, а не фиат.**  
Базовое определение:
- **1 WTR = 1 м³ воды по индексу средней стоимости (`W_m3`)**
- `W_m3` обновляется периодически (например раз в квартал), на основе публичных тарифов и водных отчётов.
Подробная формула: `docs/VODeco_Canon/12_Формула_индекса_стоимости_воды_Wm3.md`.

Важно: это **не fiat‑peg**, а **water‑peg**.

## 2) Двухуровневая модель токена
### A) VOD credits (pre‑sensor)
- Внутренняя расчётная единица для MVP.
- Используется до появления сенсоров и верификации данных воды.
- Нужна для участия, стейкинга, симуляций, UX‑сценариев.

### B) WTR (water‑token)
- Эмиссия **только после подтверждённых данных воды**.
- 1 WTR = `W_m3` (индекс средней стоимости м³).
- Предназначен для data‑anchoring и прозрачного управления водными ресурсами.

## 3) Эмиссия (строго по данным воды)
**WTR минтится только при подтверждении данных воды.**  
Механика:
1. IoT‑датчик фиксирует параметры.
2. Данные проходят верификацию (калибровка, независимая проверка, аудит).
3. Хэш и метаданные фиксируются в нодах.
4. Эмиссия: **1 WTR за 1 м³ подтверждённой воды**.

До запуска датчиков используется только VOD credits.

## 4) Фазы эволюции
1. **Pre‑sensor Credits** — VOD credits для участия и прототипов.
2. **Access & Participation** — вход, роли, базовое участие.
3. **Staking & Governance / TokenHub** — проекты, стейкинг, фильтр доверия.
4. **Data Anchoring** — верифицированные данные → хэш → on‑chain.
5. **Resource‑Linked** — смарт‑контракты объектов и распределений.

Ключевой тезис:
> **Токен — ключ к системе управления водой**, а не «вода в блокчейне».

## 5) Стабилизация и защита от спекуляций
- Протокол‑обменник торгует вокруг `W_m3`, а не свободного рынка.
- Динамический спред (покупка/выкуп) для защиты от арбитража.
- Ограничение DEX‑торговли на старте.
- Вестинг и блокировки для ранних участий.

## 6) Скидки на старте
**Скидка = вознаграждение за раннюю поддержку.**  
Формат:
- скидка доступна **в VOD credits** до запуска сенсоров;
- конвертация в WTR — **после data‑mint фазы**;
- опциональный lock‑period на ранние покупки.

## 7) Что реализовано в MVP (код)
В коде MVP используется упрощённая модель:
- фиксированный `TOTAL_SUPPLY` для демонстрации;
- скидка для демонстрации покупки;
- расчёты стейкинга и IRR на основе проектов.

Источник: `src/lib/tokenomics/calculations.ts`.

## 8) Следующие шаги (канон)
Чтобы токеномика стала единой и машино‑проверяемой:
- утвердить метод расчёта `W_m3` (индекс);
- описать IoT‑oracle pipeline и политику верификации;
- закрепить юридические формулировки utility/governance;
- реализовать механизм data‑mint на уровне смарт‑контрактов.
