# Токеномика и экономика VODeco (сведение моделей + привязка к коду)

## 1) Главный принцип
В материалах репозитория встречаются **две параллельные формализации**:

### A) MVP‑упрощение (реализовано в коде `vod-eco-mvp`)
- **База**: `1 VOD = 1 м³ питьевой воды` (мировая средняя стоимость).
- **WATER_COST_PER_M3** = 1.3 USD
- **Скидка** = 80%
- **Текущая цена** = 0.26 USD за VOD
- **TOTAL_SUPPLY** = 1,000,000,000 VOD (1 млрд)
- **APY‑таблица**: basic/project/research, зависящая от сроков.

Это закреплено в `vod-eco-mvp/src/lib/tokenomics/calculations.ts`.

### B) Tokenomics v2 (концепт‑модель в `vodeco_new/docs`)
- **База**: `1 VOD = 1 литр чистой питьевой воды` (вариант “Water”).
- Максимальная эмиссия вычисляется от объёма питьевой воды на Земле (порядок квадриллионов).
- Фазы эмиссии: Genesis → Data Anchoring → Resource‑Linked.
- Механизм стабилизации: floor/ceiling и резерв.

Это подробно раскрыто в `vodeco_new/docs/TOKENOMICS_V2*.md`.

## 2) Что рекомендуем считать “каноном” для WhitePaper
Чтобы WhitePaper был последовательным и не конфликтовал с MVP:
- **В WhitePaper описываем Tokenomics v2 как целевую модель** (фазы, data anchoring, стабилизация, лимит эмиссии).
- **MVP‑модель фиксируем как “стартовую утилитарную модель (v0)”** для прототипа интерфейсов/калькуляторов, не как финальную эмиссию.
- В интерактивной странице делаем переключатель “MVP mode / v2 mode” для калькуляторов.

## 3) Эволюционная модель токена (фазы)
Из документов (в т.ч. `архитектура и дополнения.txt`) логика фаза‑перехода выглядит так:
- **Phase 1 — Access & Participation**: токен = доступ/участие/готовность к governance.
- **Phase 2 — Staking & Governance / TokenHub**: токен = фильтрация спама, вес доверия, участие в проектах.
- **Phase 3 — Data Anchoring**: верифицированные данные → хэширование → фиксация on‑chain, появляется связь данных и объектов.
- **Phase 4 — Resource‑Linked**: токен взаимодействует со смарт‑контрактами объектов/данных/распределения.

Ключевой тезис для сайта/WhitePaper:
> Токен — это **ключ к системе управления водой**, а не “вода в блокчейне”.

## 4) Стейкинг и доходность (как сейчас в MVP)
### Реализация в API
В MVP стейкинг реализован как:
- пользователь отправляет `projectId` и `amount`;
- создаётся запись `Staking` со `status=active`, `apy = project.irr`;
- баланс кошелька уменьшается на сумму стейка;
- `Project.currentAmount` увеличивается;
- создаётся транзакция типа `staking`.

Источники:
- `vod-eco-mvp/app/api/wallet/stake/route.ts`
- `vod-eco-mvp/prisma/schema.prisma` (`Wallet`, `Transaction`, `Staking`, `Project`)

### APY/IRR
В сидах проекты содержат `irr` (например 15.5, 18.0, 20.0 и т.д.), а в метаданных продублирован `stakingAPY`.
Источник: `vod-eco-mvp/prisma/seed.ts`.

## 5) Эмиссия и supply: как не запутаться
Сейчас в репозитории одновременно существуют:
- **TOTAL_SUPPLY=1B** в MVP‑коде (удобно для прототипа интерфейсов).
- **MAX_SUPPLY ~ квадриллионы** в tokenomics v2 (реалистично для “1 литр = 1 токен”).

Рекомендация для канонической документации:
- Вынести это как явную “двухрежимность” (Prototype supply vs Target supply).
- Для интерактивных калькуляторов добавить переключатель базы: **литр / м³**.

## 6) Что ещё нужно уточнить (в будущем)
Чтобы токеномика стала единой и машинно‑проверяемой, нужно выбрать:
- единицу учёта (литр vs м³);
- итоговую модель supply;
- правила минта/бёрна в фазах Data Anchoring / Resource‑Linked;
- юридически корректный язык (utility/governance vs security).
